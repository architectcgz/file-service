# 文件服务优化点复盘

## 文档目的

这份文档不记录完整测试过程，而是聚焦本次分析中识别出的核心优化点，说明每个优化点的：

- 问题现象
- 根因分析
- 当前实现方式
- 优化思路
- 预期收益
- 后续落地建议

这类内容更适合用于个人复盘，也更方便后续提炼成简历中的“性能优化经历”。

## 一、服务端表单上传整文件进内存

### 1. 问题现象

- 小文件上传在较高并发下还能维持成功率，但延迟开始明显上升
- 大文件上传时，`file-service-app` 的内存占用持续抬高
- `5 MiB` 文件在并发提高后，`p95` 很快上升到秒级
- 压测结束后，服务内存仍然维持在较高水平，说明存在明显的堆内存压力和对象滞留

### 2. 当前实现方式

当前表单上传链路的处理方式是：

1. 在 `FileUploadCommandService` 中通过 `file.getBytes()` 一次性把文件全部读入 JVM 堆内存
2. 将整块 `byte[]` 传给 `UploadFileHashService` 计算 MD5
3. 再将同一份整块 `byte[]` 传给 `UploadStorageService`
4. 最终在 `S3StorageService` 中通过 `RequestBody.fromBytes(...)` 上传到对象存储

对应代码位置：

- `FileUploadCommandService.java`
- `UploadFileHashService.java`
- `UploadStorageService.java`
- `S3StorageService.java`

### 3. 根因分析

这个实现的问题不在“逻辑错误”，而在“热点路径的数据流设计”：

- 文件字节会完整进入堆内存
- 同一份数据会被重复用于多个阶段：校验、hash、上传
- 文件越大，单请求内存峰值越高
- 并发越大，堆内存占用越容易叠加
- 一旦进入高并发大文件场景，就会推高 GC、响应时间和内存滞留

本质上，这是一个典型的“服务端整文件中转”问题。

### 4. 优化方法

建议把上传链路从“整块内存模式”改为“流式 / 临时文件模式”。

可选方案如下：

#### 方案 A：流式读取 + 流式 hash + 文件流上传

- 不再调用 `file.getBytes()`
- 先读取文件头少量字节做魔数校验
- 将上传内容写入临时文件，或直接以流方式处理
- hash 改为基于 `InputStream` 或临时文件计算
- 上传对象存储时改为 `RequestBody.fromFile(...)`

#### 方案 B：以临时文件为中间介质

- 上传请求进入后，优先落到临时文件
- 文件类型校验读取文件头
- hash 对临时文件做流式计算
- S3 上传直接使用文件路径

#### 方案 C：保留小文件内存模式，大文件切换为文件流模式

- 例如小于某个阈值继续走内存
- 大于阈值则统一走临时文件路径
- 适合兼顾小文件吞吐和大文件稳定性

### 5. 预期收益

- 降低大文件场景下的 JVM 堆内存压力
- 降低 GC 频率和停顿时间
- 降低高并发上传下的 `p95` 和 `p99`
- 提升大文件场景下的系统稳定性
- 让服务扩容不再过度依赖堆大小

### 6. 复盘结论

这是当前最值得优先做的优化点，原因有两个：

- 收益最大，能直接改善高并发和大文件场景
- 改造路径清晰，属于可控的代码级优化，而不是抽象层面的“建议”

### 7. 简历表达方向

- 针对文件上传热点链路，定位服务端整文件进堆导致的高并发内存与延迟问题，推动上传链路从内存中转改为流式/文件流上传方案

## 二、配额检查采用“先查再更”，存在额外读库和并发窗口

### 1. 问题现象

- 上传链路中每次都会先做租户配额检查
- 高并发下，该逻辑会成为热点路径的一部分
- 本次压测中还真实打到了租户文件数上限 `10000`
- 说明配额相关逻辑不仅存在，而且已经对真实压测结论产生影响

### 2. 当前实现方式

当前配额控制流程是：

1. 在 `TenantDomainService` 中先查询租户信息和租户使用量
2. 在内存中判断：
   - 单文件是否超限
   - 总存储空间是否超限
   - 文件数量是否超限
3. 上传成功后，再在事务中调用使用量递增逻辑

这意味着它是典型的：

- 先读取当前值
- 再做判断
- 最后再更新

### 3. 根因分析

这种方式的问题主要有两个：

#### 问题一：每次上传都会多一次读库

- 上传本来就是高频路径
- 再额外增加一次读取租户使用量，会推高数据库压力

#### 问题二：存在并发竞争窗口

- 多个请求可能在同一时刻都读到“当前还没超限”
- 随后同时写入使用量
- 容易出现配额判断与最终占用不完全一致的情况

这种问题在单机场景不一定立刻爆炸，但在并发高、请求密时一定会越来越明显。

### 4. 优化方法

建议将配额控制改为“数据库原子条件更新”。

#### 优化思路

在数据库层面直接执行类似逻辑：

- 如果 `used_storage_bytes + fileSize <= maxStorageBytes`
- 并且 `used_file_count + 1 <= maxFileCount`
- 才允许一次性更新成功

也就是说，把“判断”和“递增”合并成一个原子动作。

#### 可以采用的方式

- 条件 `UPDATE`
- `UPDATE ... WHERE ...` 返回影响行数
- 如果影响行数为 `0`，说明配额不足

### 5. 预期收益

- 减少上传链路一次读库操作
- 提升数据库热点写路径效率
- 消除高并发下配额判断与真实占用之间的竞争窗口
- 提高容量判断的准确性

### 6. 复盘结论

这也是高优先级优化点，但它更偏“并发正确性 + 数据库路径优化”。

与“整文件进内存”相比：

- 它对内存改善不直接
- 但对高并发正确性和数据库效率很重要

### 7. 简历表达方向

- 优化上传链路租户配额控制逻辑，将“先查再更”改造为数据库原子条件更新，降低热点路径数据库开销并消除高并发竞争窗口

## 三、去重链路是“先查是否存在，再上传，再写元数据”

### 1. 问题现象

- 当前系统已经支持基于文件 hash 的去重
- 但在并发上传同一文件时，去重逻辑仍然可能出现重复上传对象存储的情况

### 2. 当前实现方式

当前流程是：

1. 根据文件 hash 查询 `storage_objects` 中是否已有记录
2. 如果没有，则上传文件到 S3
3. 上传成功后，写入 `storage_objects`
4. 再写入文件记录

数据库层面虽然有唯一索引兜底，但顺序上仍然是“先查、后传、再插”。

### 3. 根因分析

这种做法在并发下的问题是：

- 请求 A 查询时发现没有
- 请求 B 查询时也发现没有
- A、B 都开始上传到 S3
- 最后只有一个请求能成功插入唯一记录
- 另一个请求即使最终失败，也已经浪费了对象存储上传成本

所以数据库唯一索引只能兜住“最终一致性”，不能避免“前置浪费”。

### 4. 优化方法

建议从“查后上传”优化为“先争抢占位，再决定是否上传”。

可选方向：

#### 方案 A：基于数据库唯一键抢占

- 先尝试插入一个“处理中”状态的存储对象记录
- 插入成功的请求负责上传文件
- 插入失败的请求说明已有并发请求占位，可直接复用或等待结果

#### 方案 B：基于分布式锁 / 本地锁做 hash 级串行化

- 以 `appId + fileHash + bucketName` 作为锁 key
- 同一 hash 的上传过程串行执行

#### 方案 C：对象上传完成后补全元数据状态

- 将上传过程拆成：
  - 占位
  - 上传
  - 确认完成

### 5. 预期收益

- 避免并发上传同一文件时重复传输到对象存储
- 降低无效带宽消耗
- 降低无效 CPU 和 S3 请求成本
- 提升去重逻辑在高并发下的工程质量

### 6. 复盘结论

这是中优先级优化点。

原因是：

- 它不会像“整文件进内存”那样对所有请求都产生明显影响
- 但在高并发重复上传、秒传、资源复用类场景下非常关键

### 7. 简历表达方向

- 优化文件去重链路，识别“先查再传”在并发场景下的对象存储重复上传问题，设计基于唯一键占位/锁机制的去重增强方案

## 四、大文件优化方向不应该继续压榨 `/upload/file`，而应切到直传 / 分片直传

### 1. 问题现象

- 当前 `/upload/file` 是典型的服务端中转上传
- 文件字节必须先进入业务服务，再写对象存储
- 文件越大，这条链路越贵

### 2. 根因分析

问题不只是代码实现，而是架构职责：

- 如果大文件仍默认走业务服务
- 那么服务端就同时承担：
  - 网络接收
  - 文件校验
  - hash 计算
  - 对象存储上传
  - 元数据落库

这条链路天然重，不适合继续硬扛更高文件大小和更高并发。

### 3. 当前已有条件

当前项目里其实已经具备更优方向：

- `direct upload`
- `multipart`
- `presigned`

说明系统已经有向“客户端直传对象存储”演进的基础能力。

### 4. 优化方法

建议明确分层：

#### 小文件

- 允许继续走表单上传
- 但要先完成“流式/文件流改造”

#### 大文件

- 默认走 direct upload 或 presigned multipart
- 客户端直接上传到对象存储
- `file-service` 只负责：
  - 初始化上传任务
  - 返回上传 URL / uploadId
  - 记录进度
  - 完成确认
  - 元数据落库

### 5. 预期收益

- 显著降低业务服务带宽占用
- 显著降低业务服务 CPU 和堆内存压力
- 提升大文件场景系统上限
- 让对象存储承担它更擅长的文件写入职责

### 6. 复盘结论

这是大文件方向最重要的架构级优化点。

不是“把 `/upload/file` 再优化一点点”，而是：

- 重新划分职责边界
- 让大文件流量脱离业务服务中转链路

### 7. 简历表达方向

- 推动文件上传架构从服务端字节中转模式演进为对象存储直传/分片直传模式，降低业务服务带宽、内存和 CPU 压力

## 五、热点路径日志量偏大，存在低成本收益

### 1. 问题现象

- 上传开始、上传成功、分片上传、分片完成等操作当前大量使用 `info` 日志
- 在压测场景下，这些日志会被频繁触发

### 2. 根因分析

日志本身不是问题，但热点路径上的同步高频日志通常会带来：

- 字符串拼接和参数格式化开销
- 日志框架锁竞争
- stdout / 文件输出 I/O
- 容器日志采集附加成本

单次请求感知不明显，但在压测下会成为稳定存在的额外开销。

### 3. 优化方法

建议分级处理：

- 将高频、无强审计价值的日志从 `info` 降到 `debug`
- 对必要日志增加采样
- 保留错误日志和少量关键审计日志

### 4. 预期收益

- 低成本降低热点路径 CPU 和 I/O 消耗
- 提升压测场景下的吞吐稳定性
- 降低容器日志量

### 5. 复盘结论

这是低成本优化点。

它单独带来的收益不会像流式上传那样大，但改动小、风险低，适合作为顺手优化项。

### 6. 简历表达方向

- 对上传热点链路进行日志降噪与采样优化，降低高频请求下的无效 CPU 与 I/O 消耗

## 六、本次优化点的优先级排序

结合收益、实现难度和对当前压测结论的影响，建议优先级如下：

### 高优先级

1. 服务端表单上传改为流式 / 临时文件上传
2. 配额控制改为数据库原子条件更新

### 中优先级

3. 去重链路从“先查后传”改为“先占位再上传”
4. 大文件默认切到 direct upload / presigned multipart

### 低成本优先做

5. 热点路径日志降级或采样

## 七、后续建议落地顺序

建议按下面顺序推进：

1. 先改表单上传的内存模型
2. 再改配额控制的数据库原子更新
3. 顺手处理上传热点日志降级
4. 然后补一轮真实压测，验证内存、吞吐和延迟改善
5. 最后推进大文件默认直传和去重链路增强

## 八、适合复盘时重点记住的结论

这次最核心的结论不是“接口慢”，而是：

- 上传链路的主要问题来自数据流设计，而不是简单的线程数不够
- 大文件问题的核心不是把 JVM 调大，而是避免整文件进入业务服务堆内存
- 高并发问题不仅是性能问题，也是并发正确性问题，比如配额和去重
- 真正的大文件优化方向应该是对象存储直传，而不是继续强压服务端表单上传

## 九、简历提炼模板

后续你写简历时，可以按下面模板提炼：

### 模板一：代码级性能优化

- 识别文件上传热点链路中整文件进堆、重复字节处理带来的高并发内存与延迟问题，推动上传链路向流式/文件流处理改造

### 模板二：并发正确性优化

- 重构上传链路配额控制与去重逻辑，减少热点路径数据库读写开销，降低高并发下的竞争窗口和重复上传风险

### 模板三：架构级优化

- 推动文件上传从服务端字节中转模式向对象存储直传/分片直传模式演进，提升大文件场景下的系统稳定性和扩展能力
