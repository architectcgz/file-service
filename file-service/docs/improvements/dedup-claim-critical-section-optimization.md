# 文件去重临界区缩小优化复盘

## 背景

文件服务原来的去重方案使用分布式锁串行化同一 `appId + bucketName + fileHash` 的上传流程。

这个方案能解决并发重复上传问题，但锁的持有范围过大，锁内包含了：

1. 查询 `storage_objects`
2. 上传主文件到对象存储
3. 图片场景上传缩略图
4. 写入 `storage_objects`
5. 写入 `file_records`
6. 配额记账

当上传的是大文件，或者对象存储本身存在网络波动时，同一内容文件的并发请求会在整次上传期间被串行阻塞，导致吞吐下降、等待时间变长。

## 问题本质

问题不在“有没有锁”，而在“锁包住了最慢的远程 I/O”。

原先的锁虽然保证了正确性，但把对象存储上传这种高耗时操作也放进了临界区，带来了两个典型问题：

- 同 hash 并发请求等待时间跟着对象存储上传时长一起放大
- 文件越大，锁持有时间越长，串行化越严重

这类场景更适合使用“短占位 + 锁外上传”的方式，而不是“整条链路长锁”。

## 优化思路

这次优化采用的是数据库 claim 方案，而不是继续扩大 Redis 锁能力。

核心思路是：

1. 先用数据库抢一个很短的“上传资格”
2. 抢到资格的请求在锁外完成真正的对象上传
3. 其他并发请求不再跟着整次上传一起阻塞，只做短轮询
4. 一旦主上传者完成落库，等待中的请求直接复用已有 `storage_object`

这样临界区就从：

- 查重 + S3 上传 + 落库

缩短成：

- 插入 / 更新一条 claim 记录

## 落地方案

### 1. 新增去重占位表

新增表：`upload_dedup_claims`

用途：

- 记录当前哪一个请求拿到了某个 `(appId, fileHash, bucketName)` 的上传资格
- 通过 `expires_at` 防止异常退出后永久占位

关键字段：

- `app_id`
- `file_hash`
- `bucket_name`
- `owner_token`
- `expires_at`

主键：

- `(app_id, file_hash, bucket_name)`

### 2. 用 claim 抢占代替长时间分布式锁

在进入真正上传前：

- 先查 `storage_objects`
- 若不存在，则尝试插入 claim
- 插入冲突时，只允许接管“已过期 claim”

这样可以保证同一时刻只有一个请求负责真正上传对象。

### 3. 主上传者在锁外执行上传

拿到 claim 的请求会：

- 再次检查一次 `storage_objects`
- 若仍不存在，则执行真正的新对象上传
- 上传完成后落 `storage_objects` 和 `file_records`
- 最后释放 claim

整个对象存储上传过程不再持有全局去重锁。

### 4. 等待者只轮询结果，不阻塞整条链路

没有拿到 claim 的请求会：

- 轮询 `storage_objects`
- 一旦看到主上传者已经完成落库，就直接走秒传/复用路径
- 若长期看不到结果，则按超时失败处理

这样等待请求的开销远小于原来的长时间串行阻塞。

## 关键实现点

### 新增配置

新增上传去重配置：

- `upload.dedup.claim-lease`
- `upload.dedup.wait-timeout`
- `upload.dedup.poll-interval`

作用分别是：

- claim 租约时长
- 等待其他上传者完成的最长时长
- 轮询间隔

### 新增协调服务

新增服务：`UploadDedupCoordinatorService`

职责：

- 统一处理“已有对象 / 抢 claim / 锁外上传 / 等待复用”的协调逻辑

### 新增仓储

新增仓储：`UploadDedupClaimRepository`

职责：

- 尝试抢占 claim
- 在上传完成后释放 claim

### 接入普通文件和图片上传

普通文件上传和图片上传都切换到新的去重协调器：

- 普通文件上传：锁外上传主文件
- 图片上传：锁外上传主图，等待者在主图完成后各自上传自己的缩略图

## 真实环境验证

本次优化已在 Docker 真实环境完成验证。

### 1. 数据库迁移生效

已确认真实 PostgreSQL 中存在：

- `upload_dedup_claims`

### 2. 普通文件并发上传验证

同一新 `appId` 下并发 5 次上传相同文件：

- 5 个请求全部返回 `200`
- 返回的文件 URL 完全一致
- `storage_objects = 1`
- `reference_count = 5`
- `file_records = 5`
- `distinct_storage_object_ids = 1`
- `upload_dedup_claims` 最终残留 `0`

### 3. 图片并发上传验证

同一新 `appId` 下并发 3 次上传相同图片：

- 3 个请求全部返回 `200`
- 主图 URL 完全一致
- 每个请求各自产生自己的缩略图 URL
- `storage_objects = 1`
- `reference_count = 3`
- `file_records = 3`
- `distinct_storage_object_ids = 1`
- `upload_dedup_claims` 最终残留 `0`

## 这次优化的收益

### 1. 缩小了去重串行区

原先是“整次上传都在锁里”，现在变成“只在 claim 抢占阶段串行化”，大幅缩短临界区。

### 2. 降低了大文件场景下的并发等待成本

等待请求不再被整次对象上传阻塞，只需轮询结果并复用已上传对象。

### 3. 保留了跨实例去重正确性

claim 基于数据库主键和过期接管机制，不依赖单机内存状态，适用于多实例部署。

### 4. 避免了异常请求永久占位

通过 `expires_at`，即使主上传者异常退出，也能由后续请求重新接管上传资格。

## 风险与后续优化空间

这次优化已经明显缩小了临界区，但仍有后续空间：

### 1. claim 目前是轮询等待

优点是简单、稳定、改动小。

后续如果继续追求极致性能，可以考虑：

- Redis Pub/Sub
- 事件通知
- 条件变量 / 更细粒度的等待机制

以降低等待请求的轮询开销。

### 2. claim 租约目前是固定时长

目前采用固定 `claim-lease`，只要配置覆盖住一次完整上传即可。

后续可继续增强为：

- 上传期间续租
- 根据文件大小动态估算租约时长

这样能进一步降低超大文件或慢网络场景下 claim 误过期的概率。

### 3. 直传链路还可以继续统一去重策略

当前这次优化主要覆盖的是服务端表单上传和图片上传。

后续若继续推进，可考虑把直传完成态的去重落库策略也统一到同一套协调模型。

## 适合复盘/简历的表达方式

可以概括为：

- 将文件去重从“整上传链路长时间分布式锁”改造为“数据库短 claim + 锁外对象上传”模型，显著缩小同 hash 并发上传的临界区，降低大文件场景下的串行等待成本，并通过过期接管机制保证跨实例去重正确性与异常恢复能力。
