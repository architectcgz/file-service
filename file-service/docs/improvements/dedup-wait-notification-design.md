# 去重等待通知优化设计

## 背景

当前去重链路已经从“长时间持有分布式锁”收敛成了“数据库短 claim + 锁外上传”。

现状里，首个上传者负责：

1. 抢占 `(appId, fileHash, bucketName)` 对应的 claim
2. 执行真实上传与落库
3. 释放 claim

后续并发请求如果没有拿到 claim，会进入等待分支，当前做法是每 `pollInterval` 轮询一次数据库，检查 `storage_objects` 是否已经生成。

这套逻辑正确，但有两个问题：

1. 同一热点文件被大量并发上传时，等待方会把数据库读流量放大
2. 等待方明明只是在等“首个上传者完成或失败”，却只能通过轮询感知，效率偏低

## 目标

在不改变现有 claim 表和 `storage_objects` 作为最终事实来源的前提下，降低等待方的数据库轮询频率。

目标边界：

- 不改变 claim 抢占语义
- 不依赖单机内存态
- 不把 Redis 中的通知结果当成最终真相
- 任何通知丢失都不能影响正确性，只能影响等待效率

## 方案候选

### 方案 A：继续纯数据库轮询

做法：

- 维持当前实现
- 仅调大轮询间隔

优点：

- 实现最简单
- 没有新依赖

缺点：

- 热点文件等待方仍会持续打数据库
- 轮询间隔调大后，等待完成感知会变慢

结论：

- 只能算保守调参，不解决根问题

### 方案 B：PostgreSQL LISTEN / NOTIFY

做法：

- 首个上传者在事务成功后对某个 channel 做 `NOTIFY`
- 等待方用 `LISTEN` 阻塞等待

优点：

- 通知和数据库事务天然靠近
- 不额外依赖 Redis

缺点：

- 需要维护长连接监听
- Spring/JDBC 侧落地复杂度明显更高
- Docker 和连接池下的监听连接生命周期要单独管理
- 失败重试、断线重连、连接耗尽等边界更重

结论：

- 技术上可行，但对当前项目不是最小改动路径

### 方案 C：Redis pub/sub 通知 + 数据库最终校验

做法：

- 首个上传者在 claim 结束时向 Redis topic 发布结果事件
- 等待方订阅对应 topic，优先等通知
- 收到通知后仍然回到数据库检查 `storage_objects`
- 如果通知丢失或订阅失败，则退化为较低频的超时重查

优点：

- 项目已经有 Redis / Redisson 依赖，复用成本低
- 跨实例天然可用
- 可以显著降低等待方对数据库的热轮询
- 即便消息丢失，只要保留数据库复查，正确性仍然成立

缺点：

- pub/sub 不是持久消息，存在“先发布后订阅”的丢失窗口
- 需要补一层 fallback 兜底，不能完全依赖通知

结论：

- 在现有架构下，这是最小改动且收益最高的方案

## 推荐方案

采用“Redis pub/sub 通知 + 数据库最终校验 + 超时 fallback”的混合方案。

### 核心原则

- 数据库中的 `storage_objects` 仍然是最终事实来源
- Redis 只负责“尽快唤醒等待方”，不负责裁决上传结果
- 即便通知缺失，等待方也必须能靠 fallback 继续收敛

## 时序设计

### 场景一：首个上传者成功

1. 请求 A 抢到 claim
2. 请求 B 抢 claim 失败，进入等待
3. 请求 B 先做一次数据库检查，如果还没有 `storage_object`，则订阅对应 topic
4. 请求 A 完成对象上传和元数据事务
5. 请求 A 释放 claim，并发布 `COMPLETED` 事件
6. 请求 B 被唤醒，再查一次数据库
7. 请求 B 发现 `storage_object` 已存在，走瞬时复用路径

### 场景二：首个上传者失败

1. 请求 A 抢到 claim
2. 请求 B 抢 claim 失败，进入等待
3. 请求 A 上传或落库失败
4. 请求 A 释放 claim，并发布 `RETRY` 事件
5. 请求 B 被唤醒，先查数据库
6. 如果数据库仍无结果，则重新参与 claim 竞争

### 场景三：通知丢失

1. 请求 B 订阅失败，或消息在订阅前已发布
2. 请求 B 在一个较长的等待片段结束后重新查数据库
3. 若仍无结果，再继续 claim 竞争 / 等待

这保证了：

- 正确性不依赖消息可靠投递
- 数据库读频率从“高频轮询”下降到“事件驱动 + 低频兜底”

## 实现边界

### 新增组件

- `UploadDedupNotificationService`

职责：

- 基于 `(appId, bucketName, fileHash)` 生成通知 topic
- 发布 `COMPLETED` / `RETRY` 事件
- 让等待方阻塞等待一小段时间
- 在通知不可用时显式返回 fallback 状态

### 现有组件改造

- `UploadDedupCoordinatorService`

改动：

- claim 抢占失败后，不再只做 `Thread.sleep + 查库`
- 改为“查库一次 -> 订阅等待 -> 查库一次 -> 继续循环”
- 首个上传者在成功或失败释放 claim 后发布通知

- `UploadDedupProperties`

新增配置：

- `notification-max-wait`
  - 单次等待通知的最长时长
  - 默认建议比原来的 `pollInterval` 大一个数量级，例如 `3s`

## 配置建议

- `claim-lease`
  - 仍然覆盖完整上传时间
- `renew-interval`
  - 仍然显著小于 `claim-lease`
- `notification-max-wait`
  - 建议 `1s ~ 3s`
  - 太小会退化成频繁 fallback
  - 太大则在消息丢失时恢复较慢

## 风险与应对

### 风险一：先发布后订阅

应对：

- 等待方在进入等待前后都做数据库检查
- 即便错过消息，也能靠 fallback 收敛

### 风险二：Redis 不可用

应对：

- 通知服务失败时退化到原有低频轮询
- 不影响上传正确性

### 风险三：事件过多

应对：

- topic 粒度收敛在 `(appId, bucketName, fileHash)`
- 只发布短消息，不存历史

## 验证方式

1. 单元测试
   - 成功上传后等待方应收到通知并复用对象
   - 失败上传后等待方应被唤醒并重新竞争 claim
   - Redis 通知失败时应退化到 fallback

2. Docker 实测
   - 多并发同文件上传
   - 对比改造前后的数据库查询频率
   - 确认最终仍只有一个 `storage_object`

## 结论

推荐采用 Redis pub/sub 混合方案：

- 正确性继续由数据库保证
- 等待效率由 Redis 事件提升
- 改动面相对可控
- 与当前 Redis / Redisson 依赖最匹配
