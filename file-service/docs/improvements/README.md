## 场景一：file-gateway 访问链路返回 500

### 问题场景

- 真实访问链路为：`上传文件 -> 查询文件详情 -> 签发访问 ticket -> gateway 访问 content`
- `GET /api/v1/files/{fileId}/content` 返回 `500`
- `file-gateway` 访问同一路径时也失败，因为上游 `file-service` 无法正确处理该路由

### 根因分析

- `/content` 控制器代码实际上已经补上了
- 但在重建 Docker 镜像前，没有先执行 `mvn package`
- 导致容器里运行的仍然是旧 jar
- 旧 jar 中并不包含 `/api/v1/files/{fileId}/content`
- 所以运行时仍然报 `No static resource .../content`

### 采用的优化 / 修复方式

- 在 `file-service` 中新增专门的跳转控制器
- 复用已有的 `FileAccessService` 做文件真实访问地址解析
- 统一返回 `302 + Location + Cache-Control`
- 严格按“先 Maven 打包，再 Docker 重建”的方式重新部署
- 在真实容器环境中重新验证整条访问链路

### 相关实现位置

- `src/main/java/com/architectcgz/file/interfaces/controller/FileAccessController.java`

### 真实验证结果

- `GET file-service /api/v1/files/{fileId}/content` 返回 `302`
- `GET file-gateway /api/v1/files/{fileId}/content?ticket=...` 返回 `302`
- 网关链路压测达到约 `3358 req/s`
- `p95` 约为 `153.64 ms`
- 错误率为 `0%`

### 可用于简历的表述方向

- 优化文件访问网关链路，修复 Docker 部署下的上游重定向路由缺失问题
- 恢复基于 ticket 的文件访问跳转能力，并在真实环境下完成链路验证与压测
- 在真实压测环境中实现 `3358+ req/s`、`0%` 错误率的网关访问能力

## 场景二：服务端表单上传在高并发下出现延迟拐点

### 问题场景

- 目标接口：`POST /api/v1/upload/file`
- 小文件上传在约 `139 VUs` 时仍然比较稳定
- 到约 `219 VUs` 时虽然还没有报错，但延迟已经明显上升
- 更大的文件如 `5 MiB` 在更低并发下就出现明显的延迟恶化

### 当前压测证据

- 小文件，`139 VUs`：约 `167.85 req/s`，`p95 793.26 ms`，错误率 `0%`
- 小文件，`219 VUs`：约 `182.78 req/s`，`p95 1.33 s`，错误率 `0%`
- `1 MiB` 文件，`39 VUs`：约 `39.48 req/s`，`p95 704 ms`
- `5 MiB` 文件，`39 VUs`：约 `14.24 req/s`，`p95 2.23 s`

### 当前瓶颈分析

- 服务端表单上传当前是“整文件读入 JVM 内存”
- 然后再用同一份内存数据做 hash 计算和 S3 上传
- 大文件场景下，会显著增加堆内存占用、GC 压力和请求延迟
- 上传侧的配额检查目前是“先查再更新”，每次上传至少多一次数据库读取，并且存在并发竞争窗口
- 上传路径上的高频 `info` 日志也会增加额外 CPU 和 I/O 开销

### 计划采用的优化方式

#### 1. 将表单上传从整文件内存模式改为流式 / 临时文件上传

- 将 `file.getBytes()` 这类整块内存读取改成 `InputStream` 或临时文件路径方式
- 优先复用 `RequestBody.fromFile(...)` 这种文件流上传路径
- hash 计算改为流式计算，避免整文件常驻堆内存

#### 2. 将配额控制改为数据库原子更新

- 从“先检查再增加”改为“带条件的原子更新”
- 减少一次数据库查询
- 消除高并发下配额判断与真实占用之间的竞争窗口

#### 3. 降低高频上传日志开销

- 将热点路径上的上传日志从 `info` 降到 `debug`，或者增加采样日志
- 仅保留必要的审计日志和错误日志

#### 4. 将大文件流量切换到直传 / 分片直传

- 让文件字节直接从客户端上传到对象存储
- `file-service` 只负责元数据、权限校验和上传完成确认
- 从架构上降低服务端 CPU、内存和带宽压力

### 预期优化目标

- 降低上传接口在压测后的堆内存占用和内存滞留
- 降低高并发小文件上传的 `p95`
- 提升大文件上传场景下的吞吐稳定性
- 让系统扩容能力不再过度依赖 JVM 堆大小

### 可用于简历的表述方向

- 基于真实压测定位文件上传热点链路，识别高并发写入下的延迟拐点
- 针对服务端整文件中转上传链路，设计流式上传、原子配额控制和对象存储直传优化方案
- 推动文件上传架构从“服务端字节中转”向“元数据驱动的对象存储直传”演进

## 场景三：业务配额干扰性能结论

### 问题场景

- 一次针对租户 `blog` 的上传压测中，错误率达到了 `27.06%`
- 初看像是系统在高并发下开始失败
- 但进一步排查后发现并不是网络问题，也不是线程池或容器资源先扛不住

### 根因分析

- 失败原因是业务侧租户文件数配额已满
- 日志明确显示：`File count quota exceeded: 10044 used, limit is 10000`

### 采用的优化 / 测试方式

- 后续压测统一切换到新的租户标识
- 将“业务规则失败”和“基础设施容量瓶颈”分离分析
- 只基于干净租户环境得出容量结论

### 价值

- 避免把业务配额问题误判成系统性能瓶颈
- 保证压测结论更准确，能真实反映服务上限

### 可用于简历的表述方向

- 建立真实环境压测方法论，将业务配额限制与基础设施性能瓶颈分离诊断
- 提升性能测试结论的准确性，避免业务规则噪音干扰容量判断
