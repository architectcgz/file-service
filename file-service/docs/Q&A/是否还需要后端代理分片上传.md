# Q&A: 是否还需要保留后端代理分片上传？

## 问题

既然项目已经支持预签名直传，是否还需要保留后端代理分片上传？

这里的两个主要链路分别是：

- 后端代理分片上传：`/api/v1/multipart/*`
- 分片预签名直传：`/api/v1/direct-upload/*`

## 结论

需要保留，但不应该继续作为大文件上传的默认主路径。

更合适的策略是：

- 默认主路径：`/api/v1/direct-upload/*`
- 兼容/兜底路径：`/api/v1/multipart/*`
- 小文件直传：`/api/v1/upload/presign` + `/api/v1/upload/confirm`

也就是说，后端代理上传有保留价值，但定位应该从“主上传方案”降为“兼容方案、受控场景方案、异常兜底方案”。

## 为什么不建议再把代理上传作为默认主路径

### 1. 会让 file-service 成为流量中转点

`/api/v1/multipart/*` 的模式是：

1. 客户端把分片字节发给 `file-service`
2. `file-service` 再调用 MinIO/S3 的 `uploadPart`

对应实现：

- 控制器入口：[MultipartController.java](/home/azhi/workspace/projects/file-service/file-service/src/main/java/com/architectcgz/file/interfaces/controller/MultipartController.java#L69)
- 服务层上传分片：[MultipartUploadService.java](/home/azhi/workspace/projects/file-service/file-service/src/main/java/com/architectcgz/file/application/service/MultipartUploadService.java#L177)

这意味着每个分片都会额外经过一次服务端，带来：

- 更多服务端出口带宽消耗
- 更多连接数和线程占用
- 更高的网关和应用实例压力
- 更差的峰值扩展性

对大文件和高并发场景，这不是最优解。

### 2. 项目已经具备更合理的大文件主链路

`/api/v1/direct-upload/*` 已经支持：

- 初始化上传任务
- 获取多个分片的预签名上传 URL
- 断点续传恢复
- 前端并行上传多个 part
- 前端回传 `partNumber + etag` 完成合并

对应实现：

- 直传入口：[DirectUploadController.java](/home/azhi/workspace/projects/file-service/file-service/src/main/java/com/architectcgz/file/interfaces/controller/DirectUploadController.java#L26)
- 获取分片上传 URL：[DirectUploadService.java](/home/azhi/workspace/projects/file-service/file-service/src/main/java/com/architectcgz/file/application/service/DirectUploadService.java#L275)
- 完成直传上传：[DirectUploadService.java](/home/azhi/workspace/projects/file-service/file-service/src/main/java/com/architectcgz/file/application/service/DirectUploadService.java#L341)

既然已经有了分片预签名直传能力，大文件默认走代理上传就没有明显架构优势了。

## 为什么又不能直接删掉代理上传

### 1. 它仍然更容易统一控制

代理上传的优势是，分片数据先进入 `file-service`，所以服务端更容易统一处理：

- 权限校验
- 幂等控制
- 同分片并发保护
- 进度统计
- 审计与风控

例如当前 `/api/v1/multipart/*` 里就已经有同分片并发锁和已有 `etag` 复用逻辑：

- [MultipartUploadService.java](/home/azhi/workspace/projects/file-service/file-service/src/main/java/com/architectcgz/file/application/service/MultipartUploadService.java#L203)

这条链路对“客户端能力弱、业务想让服务端更强控”的场景仍然有意义。

### 2. 它适合兼容老客户端和简单客户端

不是所有客户端都适合处理完整的 multipart 直传流程。  
分片预签名直传要求客户端处理：

- 初始化任务
- 批量申请 part URL
- 直接请求对象存储
- 记录每个 part 的 `etag`
- 最后调用 complete

对于 Web 前端问题不大，但对一些历史客户端、嵌入式客户端、低复杂度 SDK 来说，代理上传通常更容易接入。

### 3. 某些网络环境下它是更稳的兜底方案

如果客户端到 MinIO/S3 的网络条件不稳定，或者前端无法直接访问对象存储，代理上传仍然是有效兜底：

- 内网隔离环境
- 受限企业网络
- 跨地域对象存储网络质量不稳定
- 浏览器直连对象存储受策略限制

这种情况下，保留 `/api/v1/multipart/*` 很有必要。

## 推荐的职责划分

### 1. 大文件默认方案

推荐默认走：

- `/api/v1/direct-upload/init`
- `/api/v1/direct-upload/part-urls`
- 客户端直传 part 到 MinIO/S3
- `/api/v1/direct-upload/complete`

适用场景：

- 浏览器前端
- App 客户端
- 大文件上传
- 高并发上传
- 希望减轻 `file-service` 带宽压力

### 2. 代理上传保留为兼容和兜底

推荐继续保留：

- `/api/v1/multipart/init`
- `/api/v1/multipart/{taskId}/parts/{partNumber}`
- `/api/v1/multipart/{taskId}/complete`

但建议将它的定位改为：

- 老客户端兼容
- 内网/受限网络兜底
- 服务端强管控场景
- 调试和排障备用链路

### 3. 小文件直传继续保留

对于不需要分片的小文件，继续使用：

- `/api/v1/upload/presign`
- `/api/v1/upload/confirm`

这条链路最简单，适合普通图片、文档等单文件上传。

## 具体建议

### 短期建议

短期内不建议删除 `/api/v1/multipart/*`，但建议在文档、SDK、前端接入指引里明确：

- 大文件默认推荐 `/api/v1/direct-upload/*`
- `/api/v1/multipart/*` 不是首选主路径

### 中期建议

如果未来客户端能力和网络条件都稳定，可以把 `/api/v1/multipart/*` 调整成：

- 内部使用
- 白名单客户端使用
- fallback 开关控制

而不是继续对所有前端公开为默认能力。

### 不建议现在做的事

当前阶段不建议：

- 直接删除 `/api/v1/multipart/*`
- 在没有迁移策略时强制所有客户端一次性切到直传
- 在没有监控和回退预案时下线代理链路

## 最终建议

最终推荐是：

- 保留后端代理分片上传
- 但把它降级为兼容和兜底方案
- 把分片预签名直传设为大文件默认主路径

一句话说，就是：

不是“不需要代理上传了”，而是“代理上传不该再承担主路径角色了”。
