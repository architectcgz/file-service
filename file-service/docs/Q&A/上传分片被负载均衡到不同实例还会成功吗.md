# Q&A: 上传时不同 part 被负载均衡到不同 file-service 实例，最后还会成功吗？

## 问题

现在 file-service 是多实例部署。  
如果同一个文件上传过程中的不同 part，被负载均衡打到了不同的 file-service 实例上，最终上传还能成功吗？这是如何实现的？

## 结论

会成功，但前提是这些实例共享同一套：

- 数据库
- 对象存储

在当前实现下，这件事不是靠“请求必须命中同一台机器”来保证的，而是靠：

- 上传会话状态落库
- 已上传分片状态以对象存储为准
- `complete` 阶段做最终校验
- 完成阶段通过状态竞争控制只允许一个实例真正完成

所以只要负载均衡把请求打到的是同一个环境下的 file-service 实例，而不是不同环境、不同库、不同桶，最终仍然可以成功。

## 这是如何实现的

### 1. 上传会话不保存在单机内存里

上传会话的核心状态保存在 `upload_tasks` 表，对应仓储接口和实现：

- 仓储接口：[UploadSessionRepository.java](/home/azhi/workspace/projects/file-service/file-core/file-core-ports/src/main/java/com/platform/fileservice/core/ports/repository/UploadSessionRepository.java)
- Postgres 实现：[PostgresUploadSessionRepository.java](/home/azhi/workspace/projects/file-service/file-core/file-core-adapters-db/src/main/java/com/platform/fileservice/core/adapters/db/PostgresUploadSessionRepository.java)

每次请求进来，服务端都会按 `uploadSessionId/taskId` 重新读取会话，而不是依赖某个实例本地缓存。对应入口在：

- [UploadAppService.java](/home/azhi/workspace/projects/file-service/file-core/file-core-application/src/main/java/com/platform/fileservice/core/application/service/UploadAppService.java#L500)

这意味着：

- part 1 命中实例 A
- part 2 命中实例 B
- `complete` 命中实例 C

也不会因为“上下文丢在某台机器内存里”而失败。

### 2. part 是否真的上传成功，以对象存储里的状态为准

对于分片上传，服务端并不把“哪些 part 已完成”作为单机真相保存在内存里，而是在需要时直接查询对象存储：

- 读取已上传 parts：[UploadAppService.java](/home/azhi/workspace/projects/file-service/file-core/file-core-application/src/main/java/com/platform/fileservice/core/application/service/UploadAppService.java#L759)

`complete` 前会重新调用 `loadUploadedParts(...)`，拿对象存储里已经存在的 parts 作为 authoritative state，然后再做校验：

- 分片完成入口：[UploadAppService.java](/home/azhi/workspace/projects/file-service/file-core/file-core-application/src/main/java/com/platform/fileservice/core/application/service/UploadAppService.java#L360)
- 完成前校验 part 列表：[UploadAppService.java](/home/azhi/workspace/projects/file-service/file-core/file-core-application/src/main/java/com/platform/fileservice/core/application/service/UploadAppService.java#L734)

所以即使不同 part 的上传请求走了不同实例，只要它们都写进了同一个对象存储 multipart upload，上游 file-service 最终看到的仍然是同一份分片结果。

### 3. 单文件直传完成时，也会重新读取对象元数据

对于 `PRESIGNED_SINGLE` 单文件直传，`complete` 阶段不是相信某个实例之前记住了什么，而是重新对对象存储做元数据读取：

- 单文件完成入口：[UploadAppService.java](/home/azhi/workspace/projects/file-service/file-core/file-core-application/src/main/java/com/platform/fileservice/core/application/service/UploadAppService.java#L430)

这里会重新读取对象大小、内容类型等信息，再创建正式的 `fileId` 记录。  
所以单文件直传同样不依赖某一台实例的本地状态。

### 4. `complete` 并发时，只会有一个实例真正完成

如果多个实例几乎同时收到同一个上传会话的 `complete` 请求，当前实现会先尝试把状态从 `initiated/uploading` 原子更新为 `completing`：

- 状态抢占接口：[UploadSessionRepository.java](/home/azhi/workspace/projects/file-service/file-core/file-core-ports/src/main/java/com/platform/fileservice/core/ports/repository/UploadSessionRepository.java#L25)
- DB 原子更新实现：[PostgresUploadSessionRepository.java](/home/azhi/workspace/projects/file-service/file-core/file-core-adapters-db/src/main/java/com/platform/fileservice/core/adapters/db/PostgresUploadSessionRepository.java#L172)

只有抢到 `completing` 的那个实例会继续执行真正的合并和建档。  
其他实例会进入等待逻辑，轮询会话是否已经完成：

- 等待完成逻辑：[UploadAppService.java](/home/azhi/workspace/projects/file-service/file-core/file-core-application/src/main/java/com/platform/fileservice/core/application/service/UploadAppService.java#L890)

真正完成后，再由数据库把状态更新为 `completed` 并写入 `fileId`：

- 标记完成：[PostgresUploadSessionRepository.java](/home/azhi/workspace/projects/file-service/file-core/file-core-adapters-db/src/main/java/com/platform/fileservice/core/adapters/db/PostgresUploadSessionRepository.java#L182)

这样可以避免：

- 重复合并
- 重复建档
- 并发完成导致多个 `fileId`

## 对当前几条上传链路分别意味着什么

### 1. `upload-sessions` + `DIRECT`

支持跨实例。  
不同请求落到不同实例没有问题，因为状态在数据库，对象分片状态在对象存储。

### 2. `upload-sessions` + `PRESIGNED_SINGLE`

支持跨实例。  
创建会话和 `complete` 落到不同实例也没有问题，因为 `complete` 会重新读取对象元数据。

### 3. 兼容接口 `/api/v1/direct-upload/*`

支持跨实例。  
这条链路底层已经桥接到同一套 `UploadAppService`：

- [DirectUploadCoreBridgeService.java](/home/azhi/workspace/projects/file-service/file-service/src/main/java/com/architectcgz/file/application/service/direct/bridge/DirectUploadCoreBridgeService.java)

所以它继承的是同一套“会话落库 + 对象存储为准 + 并发完成竞争控制”的能力。

### 4. 兼容接口 `/api/v1/multipart/*`

也支持不同 part 命中不同实例。  
原因是每次上传 part 都会按 `taskId` 重新加载会话，再使用同一个 `providerUploadId` 向对象存储上传分片：

- [MultipartUploadCoreBridgeService.java](/home/azhi/workspace/projects/file-service/file-service/src/main/java/com/architectcgz/file/application/service/multipart/bridge/MultipartUploadCoreBridgeService.java)

这条链路里，请求是否命中同一台实例并不是成功条件。

## 需要满足的前提

下面这些前提缺一不可：

- 所有 file-service 实例连接的是同一个数据库
- 所有 file-service 实例连接的是同一个对象存储配置
- 同一个上传任务的请求没有被打到不同环境
- 客户端始终带回同一个 `uploadSessionId` 或兼容链路里的 `taskId`

如果负载均衡把同一个上传任务打到了不同环境，例如：

- 实例 A 对应测试库
- 实例 B 对应生产库

或者：

- 实例 A 连接桶 1
- 实例 B 连接桶 2

那当然会失败，因为那已经不是“多实例共享同一状态”，而是“请求被路由到了不同系统”。

## 一句话总结

会成功。当前 file-service 的多实例上传成功条件，不是“所有请求必须打到同一台实例”，而是“所有实例共享数据库和对象存储，并且完成阶段以数据库状态和对象存储状态作为最终真相”。
